# 首次初始化引导（BOOTSTRAP）

> **若你正在维护 `mmchong` 素材库本身**：本文件面向「使用 mmchong 应用到**新业务仓库**」的场景；在本仓库内可忽略或删除根目录的 `BOOTSTRAP.md`。

本文件用于在 **首次** 将 mmchong 治理模板应用到仓库后，补齐 **真实项目信息**。全部确认并写入 `AGENTS.md`、`index.md` 等文档后，**删除本文件**，避免与长期文档重复。

---

## 1. 何时需要按本文件操作

在以下情况视为 **需要引导**，应优先完成本节流程再大规模写业务代码：

- 仓库是新项目，或刚从模板复制了 `AGENTS.example.md` / `ROOT_INDEX.example.md` 尚未改写；或
- 根目录仍存在本文件 `BOOTSTRAP.md`，且 `AGENTS.md` / `index.md` 中仍有明显占位符、与实际情况不符。

若项目已成熟、`AGENTS.md` 与 `index.md` 已反映真实栈与目录，可直接删除本文件。

---

## 2. Agent 与维护者协作（建议问题清单）

请 **Agent 或维护者** 先自行判断：若为「新项目 / 未填模板」，再 **向用户提问**（或对照已有资料），收齐后再落笔。建议覆盖：

| 主题 | 说明 |
|------|------|
| 项目定位 | 产品/库名称、一句话目标、主要用户或场景 |
| 技术栈 | 语言与版本、包管理器、前端/后端/移动端框架、数据库与 ORM、测试与构建工具 |
| 仓库布局 | 顶层目录职责（如 `apps/`、`packages/`、`src/` 等），与团队约定是否一致 |
| 质量与门禁 | 是否使用本仓库自带的 `scripts/`、pre-commit、Cursor hooks；是否需要删减或增补命令 |
| 文档约定 | 是否采用层级 `index.md`、长期记忆（如 `harness/memory/MEMORY.md`） |

将答案整理进 **`AGENTS.md`**（由 `templates/AGENTS.example.md` 复制改名或合并）与 **根目录 `index.md`**（由 `templates/ROOT_INDEX.example.md` 复制改写）。

## 3. 利用 Subagent 按层级补全 `index.md`（可选）

在根目录 **`AGENTS.md`**、**`index.md`** 已定稿或接近定稿后，若仍需为 **子目录** 批量建立 **`index.md`**（与 mmchong 约定的「每层目录说明自身 + 递归向上维护」一致），可采用 **按目录深度分层 + Subagent** 的方式，减少遗漏与重复劳动。

### 3.1 适用与原则

- **适用**：仓库已有一定目录结构（如 `apps/`、`packages/`、`src/` 多层），需要为「含业务代码、脚本或可维护文档」的目录补 `index.md`；排除 `node_modules/`、`.git/`、构建产物等无需建档目录（按团队约定）。
- **分层**：以 **目录深度**（相对仓库根）为轮次——先保证 **第 0 层**（根 `index.md`）与 **第 1 层**（各顶层目录下的 `index.md`），再依次 **第 2 层、第 3 层…**，避免子目录先写、父目录仍空白导致描述不一致。
- **Subagent**：在 Cursor 等环境中，将 **同一深度、且互不重叠的子树** 分配给不同 Subagent（并行任务）分别撰写或核对该子树内各目录的 `index.md`；**同一父目录下的多个子目录** 也可按子目录拆给多个 Subagent，由主会话约定统一模板（用途、核心文件、子项列表、关联链接）。
- **串行收口**：若多路 Subagent 会改到 **同一父级 `index.md`** 或存在交叉引用，由 **主会话** 在一层结束后合并、去重并核对与 `AGENTS.md` 是否冲突。

### 3.2 建议操作顺序（示例）

1. **主会话** 列出本轮目标深度及目录清单（可基于 `find` / 目录树，排除忽略路径）。
2. **深度 = 1**：对每个顶层目录（如 `apps/`、`packages/`），可各派 **一个 Subagent**，任务为：为该目录及其 **直接** 子结构写/补 **`该目录/index.md`**（内容聚焦本层职责与下一层入口，不必一次写完所有深层）。
3. **深度 = 2, 3, …**：对上一层 `index.md` 已覆盖的路径，继续向下层派发 Subagent；每层结束由主会话检查父级索引是否需同步一句「子目录详见 xxx/index.md」。
4. **每份 `index.md` 建议包含**：本目录用途、核心文件与子目录列表、与上级 `index.md` / `AGENTS.md` 的呼应、可选「关联」小节。
5. **与门禁对齐**：若仓库启用了 pre-commit 等对「变更文件旁需有 `index.md`」的检查，Subagent 补档后应 **随代码/文档同批提交**，避免只加空壳。

### 3.3 收尾

层级索引补全后，在根 **`index.md`** 中确认一级目录表与实际情况一致；必要时更新 **`AGENTS.md`** 中的目录结构小节。再进入下一节「完成标准」。

---

## 4. 完成标准与收尾

- [ ] `AGENTS.md` 中项目名、概述、技术栈、目录结构、构建/验证方式已与当前仓库一致，无模板占位语。
- [ ] 根目录 `index.md` 已同步为一级目录与关联关系。
- [ ] 若采用层级 `index.md`：关键目录已具备 `index.md`，且与第 3 节 Subagent 分层策略一致、无大面积空白父目录。
- [ ] 按需补充或删减 `scripts/hooks/pre-commit`、依赖安装方式（如 `npm install` / `pnpm install`）等说明。
- [ ] 上述确认无误后，**删除本文件 `BOOTSTRAP.md`**，并在 `AGENTS.md` 中保留或删除「首次初始化请读 BOOTSTRAP」的提示行（建议删除该提示行，避免指向已不存在的文件）。
